## Phase II - Design Document and First Check-ins/ Scrum Meeting 
 
Instructions: 

User Interface Specification: This should include (a) a Navigation Diagram (See an example hereLinks to an external site. where nodes represent the different screens and arrows represent how to navigate from one screen to another), and (b) your screen layouts with their descriptions. Further details of this requirement are listed in the prompt for the User Interface Specification section on the GitHub repository project template.
A class diagram of your project (using UML)
A description of the class diagram
 

Additionally, you will need to:

Create an “Epic” (note) for each feature (in the Product Backlog column).
Conduct your first sprint planning meeting to plan out the next 7 days of work.
Break down the “Epics” into smaller and actionable user stories in the “Sprint Backlog” column. Convert them into issues, and assign them to the appropriate team member.
These cards should represent roughly 7 days worth of development time for your team. Then, once the sprint is over you should be repeating these steps to plan a new sprint, taking you until your second scrum meeting with the reader in phase III.
Each team member needs to submit the “Peer Evaluation Form” on Canvas for phase II. In this form, you need to fill in the names of all team members, the percentage of work contributed by each member for phase II, and a description of their contributions. Remember that each team member should submit the form individually.
Schedule two check-ins using Calendly. The scheduled time slots should both be during your lab on week 6. Your entire team must be present for both check-ins.The first check-in needs to be scheduled with your lab TA. During that meeting, you will discuss your project design/class diagram from phase II.The second check-in should be scheduled with a reader. During that meeting you will discuss:
The tasks you are planning for the first sprint
How work will be divided between the team members.
